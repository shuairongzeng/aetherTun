# Aether 基础代理配置 GUI 设计文档

## 概述

本设计的目标是在现有 GUI MVP 上补齐“新手可直接修改基础代理配置”的能力，让用户无需手工编辑 `%LOCALAPPDATA%\Aether\config.json`，也能完成最常见的接入操作。第一版严格聚焦 `proxy.host`、`proxy.port`、`proxy.type` 三个字段，只解决“把代理地址填对、保存成功、知道何时生效”这一个核心问题。`tun`、`dns`、`routing` 等高级配置继续保留在配置文件中，不在当前 GUI 暴露，避免把简单问题做复杂。

本设计沿用当前运行模型：GUI 仍然是控制层，实际运行时配置仍然保存在 `%LOCALAPPDATA%\Aether\config.json`。这意味着 GUI 中显示的内容必须对应“当前真实配置”，而不是一份额外的内存副本或专用 GUI 配置。首版不引入新的设置数据库、不引入多窗口配置器，也不引入复杂表单框架；只在主窗口内增加一个“基础代理配置”区域，通过现有 Wails 绑定与 `internal/config` 读写配置。

## 目标与非目标

### 目标

- 在主窗口内直接编辑基础代理配置，不再要求新手先打开记事本。
- 让用户一眼看懂当前配置、修改状态、保存结果，以及是否需要重启代理。
- 保存时只覆盖 `proxy.host`、`proxy.port`、`proxy.type`，其余配置保持原样。
- 对非法输入提供字段级提示，减少“点了按钮但不知道错在哪”的困惑。

### 非目标

- 不在本阶段提供 `tun`、`dns`、`routing` 的 GUI 编辑能力。
- 不提供完整 JSON 编辑器、导入导出、配置历史版本、配置对比。
- 不引入新的模态框系统；“立即重启以应用配置”可先使用轻量确认交互实现。

## 页面结构与信息架构

推荐在主窗口中新增一个全宽的 `BasicProxyConfigCard`，位置放在 `PrimaryActionCard` 下方、日志区上方。这样用户的阅读顺序自然变成：先看运行状态，再进行启动/停止操作，然后查看或修改代理配置，最后再去看日志。相比弹窗或独立配置窗口，这种布局更稳定，也更符合当前“单窗口 + 托盘”的产品心智。

配置卡片仅包含三项表单字段和三类动作。字段分别是：

- `代理地址`：文本输入框，对应 `proxy.host`
- `代理端口`：数字输入框，对应 `proxy.port`
- `代理类型`：下拉框，对应 `proxy.type`，第一版支持 `SOCKS5` 与 `HTTP`

动作分别是：

- `保存配置`：仅在表单有修改且通过校验时可点击
- `恢复默认值`：恢复到 `config.DefaultConfig().Proxy`
- `打开配置文件`：继续保留给高级用户的兜底入口

`QuickActionsCard` 仍可保留，但“打开配置文件”既可以在快捷操作区出现，也可以在配置卡片中重复提供；对于新手来说，在修改配置的上下文中直接提供“打开配置文件”更容易理解。

## 交互规则

页面加载时，GUI 通过后端接口读取 `%LOCALAPPDATA%\Aether\config.json` 的当前值，并将其作为表单初始值。如果配置文件不存在，则沿用当前行为自动创建默认配置后返回结果。这样可以保证 GUI 看到的始终是实际生效文件，而不是另一套仅供显示的状态。

表单交互遵循“未改不打扰、错误就地提示、保存状态清晰”三项规则：

- 用户未做任何修改时，`保存配置` 按钮禁用
- 用户修改后若存在非法值，按钮保持禁用，并在字段下方显示中文错误提示
- 用户点击保存后，按钮进入 `保存中...` 状态，避免重复提交
- 保存成功后给明确成功反馈，而不是静默刷新

校验规则尽量直白：

- `代理地址` 不能为空
- `代理端口` 必须是 `1-65535`
- `代理类型` 必须属于支持列表

如果代理当前未运行，保存成功后即可结束；如果代理正在运行，则在保存成功后立刻提示：`新配置已保存，是否立即重启代理以应用新配置？` 用户确认后执行“停止 -> 启动”；若取消，则保留当前运行状态，并明确提示“新配置将在下次启动后生效”。

## 后端接口与数据流

后端不需要引入新的配置子系统，只需在现有 Wails 绑定层增加两类最小接口：

- `GetBasicProxySettings()`：返回基础代理配置 DTO
- `SaveBasicProxySettings(input)`：校验并保存基础代理配置，返回保存结果与是否建议重启

实现时应复用 `config.LoadOrCreate()` 与 `config.Save()`。保存流程建议是：

1. 读取当前完整配置
2. 仅覆盖 `Proxy.Host`、`Proxy.Port`、`Proxy.Type`
3. 保留 `Tun`、`DNS`、`Routing`、`LogLevel` 原值
4. 写回 `%LOCALAPPDATA%\Aether\config.json`

为避免前端自己猜测配置是否需要重启，`SaveBasicProxySettings` 可以直接返回一个结果对象，例如：

- `settings`：归一化后的最新值
- `requiresRestart`：保存时代理是否处于运行中

前端据此决定是否弹出“立即重启”确认，并在重启成功后调用现有状态轮询刷新 UI。

## 错误处理

错误提示需要区分层级，避免用户把“保存失败”和“重启失败”混为一谈：

- **字段校验错误**：直接显示在输入框下方，例如“端口必须在 1-65535”
- **配置读取失败**：显示顶层错误，并保留“打开配置文件”入口
- **配置文件损坏 / JSON 非法**：提示“配置文件格式异常”，不要无声覆盖用户文件
- **保存成功但重启失败**：明确显示“配置已保存，但代理重启失败”，避免用户误认为配置没有保存

第一版不建议在保存失败时自动回滚输入框内容。用户应保留已填写的表单，便于修正后再次提交。

## 测试策略

测试分为三层：

1. **Go 单元测试**
   - 验证基础代理字段校验逻辑
   - 验证只更新 `proxy.*` 时不会覆盖 `tun/dns/routing`
   - 验证 Wails 绑定读写正确指向 `%LOCALAPPDATA%\Aether\config.json`

2. **前端组件测试**
   - 验证初始值渲染
   - 验证脏状态与按钮禁用逻辑
   - 验证字段级错误提示
   - 验证保存中状态与成功提示

3. **前端集成测试**
   - 验证保存成功后，运行中场景会出现“立即重启”确认
   - 验证用户确认后触发停止与启动流程
   - 验证取消重启时仍保留“配置将在重启后生效”的提示

## 结论

这次改动是 GUI MVP 从“只会启停”走向“新手可配置”的关键一步，但仍然坚持 YAGNI：只做最小可用的基础代理配置，不把高级网络参数一股脑推给用户。设计上优先保证真实配置来源、保存反馈清晰、运行中改配置有明确生效路径。完成后，用户可以在不离开主界面的情况下完成“填入上游代理 -> 保存 -> 启动代理”的完整闭环，这将显著降低首用门槛，也为后续首启向导和连接测试功能打下稳定基础。
