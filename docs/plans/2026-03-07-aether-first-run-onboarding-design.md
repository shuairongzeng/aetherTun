# Aether 首启向导设计文档

## 概述

本设计的目标是在现有 GUI 与基础代理配置能力之上，再补上一层真正面向新手的“首次启动引导”。用户第一次打开 `Aether.exe` 时，不需要先理解配置文件路径、JSON 字段或快捷操作区，而是先通过一个非常轻量的引导层完成最基础的代理接入。第一版严格控制范围：允许跳过、不做连接测试、不引入单独窗口、不增加复杂的步骤管理，只解决“第一次打开时把用户带到正确入口”这一件事。

我们已经确认该向导的产品规则如下：

- 向导 **可以跳过**
- 如果用户跳过，主界面会持续提示“尚未完成首次配置”
- 触发条件为：`%LOCALAPPDATA%\Aether\config.json` 不存在，或其基础代理配置仍等于默认值
- 第一版只收集 `proxy.host`、`proxy.port`、`proxy.type`
- 第一版 **不做连接测试**

这意味着首启向导不是一套新的配置系统，而是现有基础代理配置能力的“引导入口层”。它应复用当前的真实配置文件和保存逻辑，避免一套 UI 写入临时状态，另一套 UI 再去落盘，造成维护负担。

## 目标与非目标

### 目标

- 在疑似首启场景下，自动把用户带到基础代理配置入口
- 允许跳过，但在主界面保留明显、可再次进入的提醒
- 保存成功后立即关闭引导层，并与主界面的配置卡片保持一致
- 整个体验保持轻量，不增加新的窗口和复杂步骤系统

### 非目标

- 不实现代理可达性检测、TCP 测试、代理握手测试
- 不做安装向导式的多页流程
- 不单独保存“首启完成标志位”文件
- 不在这一版暴露 `tun`、`dns`、`routing` 等高级配置

## 信息架构与页面形态

推荐将首启向导实现为 **主界面上的覆盖式引导层**，而不是弹窗或独立窗口。启动后主界面仍然正常渲染，但在满足触发条件时，界面上方覆盖一张大卡片，阻止用户一上来直接误点“启动代理”，同时保留与现有界面的连续性。完成配置后，只需要关闭覆盖层即可，无需额外跳转。

第一版向导采用 **两步结构**：

1. **欢迎页**
   - 说明 Aether 需要一个可用的上游代理
   - 告诉用户先完成基础配置，再启动代理
   - 提供 `开始配置` 与 `暂时跳过`

2. **基础代理配置页**
   - 输入 `代理地址`
   - 输入 `代理端口`
   - 选择 `代理类型`
   - 按钮为 `保存并进入主界面`

相比一步直接塞满表单，两步结构更容易理解；相比多步安装向导，又足够轻。用户如果点 `暂时跳过`，则关闭覆盖层，但主界面顶部会出现一个明显的提醒条，文案类似：

`尚未完成首次代理配置，建议先填写代理地址、端口和类型，再启动代理。`

提醒条需要提供两个动作：

- `继续配置`
- `打开配置文件`

这样可以保证跳过并不意味着“用户彻底失联”，而是可以非常轻松地回到正确入口。

## 触发与关闭规则

首启向导不依赖额外的布尔标记文件，而是直接依据真实配置状态判断是否展示，这样逻辑更稳，也避免“状态文件和实际配置脱节”。判断逻辑建议是：

1. 读取 `%LOCALAPPDATA%\Aether\config.json`
2. 如果文件不存在，返回 `shouldShowOnboarding = true`
3. 如果文件存在，但 `proxy.host`、`proxy.port`、`proxy.type` 与 `config.DefaultConfig().Proxy` 完全一致，也返回 `true`
4. 其他情况返回 `false`

关闭规则同样保持简单：

- **保存成功**：关闭向导，不再显示提醒条
- **点击跳过**：只关闭当前覆盖层，但保留主界面提醒条
- **关闭应用后再次启动**：若配置仍是默认值，则继续显示向导
- **用户后来在主界面配置卡片中完成配置**：提醒条自动消失

这样的设计保证首启提示与“真实是否已经配好”保持同步，不会出现“其实已经改了配置，但向导/提醒还一直纠缠”的问题。

## 后端接口与数据流

后端建议只增加一个新的最小能力：`GetOnboardingState()`。它返回一个轻量 DTO，例如：

- `shouldShowOnboarding`
- `isDefaultProxyConfig`
- `configExists`

保存动作无需新增专用 onboarding API，而是直接复用现有的 `SaveBasicProxySettings()`。这样数据流会非常清晰：

1. App 初始化
2. 前端请求 `GetOnboardingState()`
3. 若需要展示，则渲染向导覆盖层
4. 用户在向导中填写基础代理配置
5. 前端调用现有 `SaveBasicProxySettings()`
6. 保存成功后关闭向导，并同步更新主界面配置卡片

跳过时，前端只维护一个当前会话内的 “dismissed” 状态，用于本次隐藏覆盖层；但是否继续显示提醒条，仍由后端返回的真实 onboarding 状态决定。

## 测试策略

### Go 测试

- 无配置文件时，`GetOnboardingState()` 返回应显示向导
- 默认代理配置时，返回应显示向导
- 自定义基础代理配置时，返回不显示向导

### 前端组件测试

- 覆盖式欢迎页正确渲染
- 点击 `暂时跳过` 后关闭覆盖层并显示提醒条
- 点击 `继续配置` 后重新打开向导

### 前端集成测试

- 首次进入时展示向导
- 向导保存成功后关闭，且提醒条消失
- 跳过后提醒条保留，主界面仍可继续操作

## 结论

这版首启向导的价值不在于功能多，而在于把正确路径提前：让第一次使用 Aether 的用户在最开始就理解“先填上游代理，再启动代理”。设计上坚持最小可行原则：不新建窗口、不做连接测试、不引入额外持久化状态，而是完全依托真实配置状态与现有基础代理配置能力完成引导。这样既能显著降低新手门槛，又不会让产品迅速膨胀成一套沉重的安装/向导系统，也为后续如果需要加入“连接测试”或“更多配置项”预留了清晰扩展点。
